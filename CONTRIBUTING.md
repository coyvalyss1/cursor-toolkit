# Contributing to Cursor Toolkit

Thanks for your interest in contributing! We welcome contributions from the community.

## How to Contribute

### Found a Bug?

- Check if it's already been reported in [Issues](https://github.com/coyvalyss1/cursor-toolkit/issues)
- If not, open a new issue with:
  - Clear description of the problem
  - Steps to reproduce
  - Expected vs actual behavior
  - Your Cursor version and OS

### Have a Rule or Skill to Share?

We love seeing how others are using Cursor rules and skills!

**Before submitting:**
1. Make sure your rule/skill is generic enough for others to use
2. Remove any project-specific references (API keys, domains, internal names)
3. Include clear examples and comments
4. Test it with Cursor to make sure it works

**To submit:**
1. Fork the repo
2. Create a new branch (`git checkout -b feature/your-rule-name`)
3. Add your rule/skill to the appropriate folder
4. Update README.md if needed
5. Commit your changes (`git commit -m 'Add: [description]'`)
6. Push to your fork (`git push origin feature/your-rule-name`)
7. Open a Pull Request

### Improving Documentation

Found a typo? Have a clearer way to explain something? Documentation PRs are always welcome.

## Code of Conduct

- Be respectful and constructive
- Focus on the idea, not the person
- Assume good intent
- Keep discussions on-topic

## What We're Looking For

**High priority:**
- Rules for popular frameworks (Next.js, Vue, Angular, etc.)
- Best practices backed by research (with citations)
- Skills for common multi-step workflows
- Language-specific patterns (Python, Go, Rust, etc.)

**Not a good fit:**
- Highly project-specific rules without clear generalization
- Rules that duplicate existing functionality
- Anything that leaks credentials, API keys, or private info

## Questions?

Open an issue with the "question" label, or reach out to [@coybyron](https://github.com/coyvalyss1).
